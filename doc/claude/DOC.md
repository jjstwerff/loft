# Documentation Generation

## How it works

Documentation is generated from two sources and written to `doc/`.
Run `cargo run --bin gendoc` to regenerate everything.

### `doc/<stem>.html` — language topic pages

Generated by `src/documentation.rs`, orchestrated by `src/gendoc.rs`.

Each `.loft` file in `tests/docs/` produces one HTML page. The file is split into alternating **prose** (consecutive `//` comment lines) and **code** (consecutive non-comment lines) sections. Blank lines end each section. Prose becomes `<p>` tags; code becomes `<pre><code>` with syntax highlighting.

The page title and short name come from two header comments at the top of the file:

```
// @NAME: Lexer
// @TITLE: Lexer library.
```

`@NAME` appears in the navigation bar and as the `<h1>`. `@TITLE` appears on the index card grid.

The first prose block after the header comments becomes the page introduction.

Files with a `00-` prefix are excluded from page generation. `doc/00-vs-rust.html` is maintained by hand. **Important:** when new topic pages are added (new numbered `.loft` files), the nav bar inside `doc/00-vs-rust.html` must be updated manually to include the new links — it is not regenerated automatically.

### `doc/stdlib-<id>.html` — standard library section pages

Parsed by `src/gendoc.rs`, rendered by `src/documentation.rs`.

Reads `default/01_code.loft`, `default/02_images.loft`, `default/03_text.loft`. Public declarations (`pub fn`, `pub struct`, `pub enum`) and their preceding `//` doc comments are extracted and grouped by `// --- Section Name ---` headers. Functions show only the signature; structs and enums include the full body.

Each section becomes its own file: `doc/stdlib-types.html`, `doc/stdlib-math.html`, `doc/stdlib-text.html`, `doc/stdlib-collections.html`, `doc/stdlib-output-and-diagnostics.html`, `doc/stdlib-file-system.html`, `doc/stdlib-environment.html`.

`doc/stdlib.html` is a card-grid overview linking to all section pages.

### Stdlib links

Links from highlighted identifiers to their stdlib section page are added **inline during syntax highlighting** — there is no separate post-processing pass. `highlight_loft` accepts a `link_map: &HashMap<String, String>` that maps identifier names to URLs like `stdlib-output-and-diagnostics.html`. The map is built in `gendoc.rs` before any HTML is written.

### Navigation

Every page (language topics and stdlib sections) shares the same nav structure built by `build_nav` in `src/documentation.rs`:

```
Home · Language: vs Rust · Keywords · … · Library: Types · Math · … [search input]
```

- **Home** links to `index.html`.
- **Language** lists all suite topic pages (excluding `00-` prefix files).
- **Library** lists all stdlib section pages.
- The current page is shown bold without a link.

### Search

`doc/search-index.js` is regenerated on every `gendoc` run. It contains a `SEARCH_INDEX` array with entries for every stdlib section, stdlib item (function, type, struct, enum), and language topic.

`doc/search.js` is a static file. It reads `SEARCH_INDEX` and filters it on the search input, showing up to 10 results in a dropdown.

### Index page

`doc/index.html` shows two card grids: one for language topics (from the suite files) and one for standard library sections. It is regenerated by `generate_docs` inside `src/documentation.rs`.

**Intro paragraph grouping:** The helper `index_intro` reads the `00-general.loft` comment blocks and groups consecutive non-blank comment lines into `<p>` elements. A plain blank line or a bare `//` line triggers a paragraph break. The helper `flush_intro_para` flushes the accumulated buffer. This ensures the intro renders as readable paragraphs rather than one line per `<p>` tag.

**Search bar on index:** A `<div class="search-wrap index-search">` with `<input id="search">` and `<div id="search-results">` is emitted inside `<header class="index-header">`. The already-loaded `search.js` finds these elements by id and makes the search functional on the index page.

**Stdlib card descriptions:** `StdlibSection` has a `description: String` field (in addition to `id` and `name`). `gendoc.rs` populates it via `stdlib_description(id)`, a match on known section slugs returning hard-coded one-line descriptions. `write_index` emits `<p>description</p>` inside each stdlib card when the field is non-empty.

**Index page CSS:** `doc/style.css` defines three classes used by the index header:
- `.index-header` — flex column, centers hero + search
- `.index-hero` — the `<h1>` + tagline block
- `.index-search` — constrains the search input to `max-width: 400px` and uses `width: 100%`

## Source file roles

| File | Role |
|---|---|
| `src/gendoc.rs` | Binary entry point (`cargo run --bin gendoc`). Reads the default library files and suite `.loft` files, extracts public declarations and doc comments, builds the `link_map`, then calls functions in `src/documentation.rs` to emit HTML. |
| `src/documentation.rs` | HTML generation library. Provides `generate_docs` (index + topic pages), `gather_topic_info`, `build_nav`, `page_html`, and `highlight_loft` (syntax highlighting with inline stdlib links). |

`gendoc.rs` is the **orchestrator** — it owns the data-gathering loop and calls `documentation.rs` for all HTML emission. `documentation.rs` contains no file I/O of its own beyond writing the output files it is explicitly told to write.

## Syntax highlighting classes

| Class | Meaning |
|---|---|
| `kw` | Language keyword (`fn`, `if`, `for`, …) |
| `ty` | Primitive type (`integer`, `text`, `vector`, …) |
| `bi` | Built-in function (`assert`, `panic`, `len`, …) |
| `en` | Capitalised identifier (struct/enum name) |
| `fn-call` | Identifier followed by `(` |
| `st` | String literal |
| `ch` | Character literal |
| `nm` | Numeric literal |
| `cm` | Line comment |

Identifiers whose name appears in the link map emit a `<a href="…">` inside the span automatically.

## Adding a new topic page

1. Create `tests/docs/NN-name.loft` with `@NAME` and `@TITLE` headers.
2. Write comment blocks before each code group to explain what the code demonstrates.
3. Run `cargo run --bin gendoc` to regenerate all HTML files.

The new page is picked up automatically; the nav and index card grid are rebuilt from the full directory listing on every run.

## Bundled library documentation

Libraries extracted from the stdlib (imaging, random) and third-party packages
live in `lib/<name>/`.  Their documentation is generated alongside the main
docs and linked from the index page.

### How it works

`gendoc.rs` discovers all `lib/*/loft.toml` manifests after generating the
main docs.  For each package it calls `generate_pkg_docs()` which:

1. Reads `loft.toml` for the package name and version
2. Extracts public declarations and doc comments from `src/*.loft`
3. Reads guide pages from `docs/*.loft` (same `@NAME`/`@TITLE` format as topic pages)
4. Writes self-contained HTML to `lib/<name>/doc/`

### Index integration

The main `docs/index.html` has three card grids:

| Section | Source | Example cards |
|---------|--------|---------------|
| **Language** | `tests/docs/*.loft` | Keywords, Control Flow, Match, … |
| **Standard Library** | `default/*.loft` | Types, Math, Text, File System, … |
| **Libraries** | `lib/*/loft.toml` | Imaging, Random, Graphics, … |

Each library card links to `lib/<name>/doc/index.html`.  The link is relative
so it works both on the GitHub Pages site and when browsing locally.

### Navigation

The shared nav bar (`build_nav`) includes a **Libraries** section after the
stdlib sections.  Library names are auto-discovered from `lib/*/loft.toml`.

### Search

Library types and functions are merged into the main `search-index.js` so
users can find `Image`, `Pixel`, `rand`, `rand_seed` from the global search.

### Adding a new bundled library

1. Create `lib/<name>/loft.toml` with `[package] name = "..."`.
2. Write `src/<name>.loft` with `pub` declarations and `//` doc comments.
3. Optionally add guide pages in `docs/*.loft`.
4. Run `cargo run --bin gendoc` — the library is auto-discovered and its docs
   are generated and linked from the index.

### Two tiers of library documentation

| Tier | Where | How documented | Example |
|------|-------|----------------|---------|
| **Bundled** (2–3 packages) | `lib/` in interpreter repo | Generated locally by `gendoc`; linked from index | imaging, random |
| **Registry** (40+ packages) | Independent GitHub repos | Catalog page with links to external docs | graphics, server, web |

**Bundled libraries** are extracted from the stdlib and ship with the
interpreter.  They always work without `loft install`.  Their docs are
generated by `gendoc` and appear as cards on the index page, in the nav
bar, and in the search index.

**Registry libraries** are independently maintained.  The main docs site
generates a **Package Catalog** page from `registry.txt` (the same file
used by `loft registry sync`).  Each entry shows the package name, version,
one-line description, and a link to the package's own hosted docs.

The registry file format (`name version url`) gains an optional `docs` field:

```
# name  version  url                                          docs
graphics 0.2.0  https://example.com/graphics-0.2.0.zip       https://jjstwerff.github.io/loft-graphics/
server   0.1.0  https://example.com/server-0.1.0.zip         https://jjstwerff.github.io/loft-server/
```

The catalog page is a simple table generated from this data — no packages
need to be installed to show them.  Each package maintains its own docs
(generated by `loft doc` and hosted on GitHub Pages or similar).

### User discoverability

Users find libraries through four paths:

1. **Index page** — bundled libraries appear as cards alongside stdlib
2. **Search** — bundled library types and functions are in the global search
3. **Nav bar** — Libraries section on every page
4. **Package Catalog** — linked from the index; lists all registry packages
   with descriptions and external doc links

This scales from 3 bundled libraries to 100+ registry packages without
bloating the interpreter repo or the main docs build.

## Adding documentation to the standard library

Add `//` doc comments immediately above `pub` declarations in `default/*.loft`.
Use `// --- Section Name ---` lines to group related items under a heading.
Run `cargo run --bin gendoc` to regenerate the section pages and update the link map.

If a new section is added, a new `doc/stdlib-<id>.html` file is created automatically and the section appears in the nav and search index on the next run.
