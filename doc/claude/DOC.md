# Documentation Generation

## How it works

Documentation is generated from two sources and written to `doc/`.
Run `cargo run --bin gendoc` to regenerate everything.

### `doc/<stem>.html` â€” language topic pages

Generated by `src/documentation.rs`, orchestrated by `src/gendoc.rs`.

Each `.loft` file in `tests/docs/` produces one HTML page. The file is split into alternating **prose** (consecutive `//` comment lines) and **code** (consecutive non-comment lines) sections. Blank lines end each section. Prose becomes `<p>` tags; code becomes `<pre><code>` with syntax highlighting.

The page title and short name come from two header comments at the top of the file:

```
// @NAME: Lexer
// @TITLE: Lexer library.
```

`@NAME` appears in the navigation bar and as the `<h1>`. `@TITLE` appears on the index card grid.

The first prose block after the header comments becomes the page introduction.

Files with a `00-` prefix are excluded from page generation. `doc/00-vs-rust.html` is maintained by hand. **Important:** when new topic pages are added (new numbered `.loft` files), the nav bar inside `doc/00-vs-rust.html` must be updated manually to include the new links â€” it is not regenerated automatically.

### `doc/stdlib-<id>.html` â€” standard library section pages

Parsed by `src/gendoc.rs`, rendered by `src/documentation.rs`.

Reads `default/01_code.loft`, `default/02_images.loft`, `default/03_text.loft`. Public declarations (`pub fn`, `pub struct`, `pub enum`) and their preceding `//` doc comments are extracted and grouped by `// --- Section Name ---` headers. Functions show only the signature; structs and enums include the full body.

Each section becomes its own file: `doc/stdlib-types.html`, `doc/stdlib-math.html`, `doc/stdlib-text.html`, `doc/stdlib-collections.html`, `doc/stdlib-output-and-diagnostics.html`, `doc/stdlib-file-system.html`, `doc/stdlib-environment.html`.

`doc/stdlib.html` is a card-grid overview linking to all section pages.

### Stdlib links

Links from highlighted identifiers to their stdlib section page are added **inline during syntax highlighting** â€” there is no separate post-processing pass. `highlight_loft` accepts a `link_map: &HashMap<String, String>` that maps identifier names to URLs like `stdlib-output-and-diagnostics.html`. The map is built in `gendoc.rs` before any HTML is written.

### Navigation

Every page (language topics and stdlib sections) shares the same nav structure built by `build_nav` in `src/documentation.rs`:

```
Home Â· Language: vs Rust Â· Keywords Â· â€¦ Â· Library: Types Â· Math Â· â€¦ [search input]
```

- **Home** links to `index.html`.
- **Language** lists all suite topic pages (excluding `00-` prefix files).
- **Library** lists all stdlib section pages.
- The current page is shown bold without a link.

### Search

`doc/search-index.js` is regenerated on every `gendoc` run. It contains a `SEARCH_INDEX` array with entries for every stdlib section, stdlib item (function, type, struct, enum), and language topic.

`doc/search.js` is a static file. It reads `SEARCH_INDEX` and filters it on the search input, showing up to 10 results in a dropdown.

### Index page

`doc/index.html` shows two card grids: one for language topics (from the suite files) and one for standard library sections. It is regenerated by `generate_docs` inside `src/documentation.rs`.

**Intro paragraph grouping:** The helper `index_intro` reads the `00-general.loft` comment blocks and groups consecutive non-blank comment lines into `<p>` elements. A plain blank line or a bare `//` line triggers a paragraph break. The helper `flush_intro_para` flushes the accumulated buffer. This ensures the intro renders as readable paragraphs rather than one line per `<p>` tag.

**Search bar on index:** A `<div class="search-wrap index-search">` with `<input id="search">` and `<div id="search-results">` is emitted inside `<header class="index-header">`. The already-loaded `search.js` finds these elements by id and makes the search functional on the index page.

**Stdlib card descriptions:** `StdlibSection` has a `description: String` field (in addition to `id` and `name`). `gendoc.rs` populates it via `stdlib_description(id)`, a match on known section slugs returning hard-coded one-line descriptions. `write_index` emits `<p>description</p>` inside each stdlib card when the field is non-empty.

**Index page CSS:** `doc/style.css` defines three classes used by the index header:
- `.index-header` â€” flex column, centers hero + search
- `.index-hero` â€” the `<h1>` + tagline block
- `.index-search` â€” constrains the search input to `max-width: 400px` and uses `width: 100%`

## Source file roles

| File | Role |
|---|---|
| `src/gendoc.rs` | Binary entry point (`cargo run --bin gendoc`). Reads the default library files and suite `.loft` files, extracts public declarations and doc comments, builds the `link_map`, then calls functions in `src/documentation.rs` to emit HTML. |
| `src/documentation.rs` | HTML generation library. Provides `generate_docs` (index + topic pages), `gather_topic_info`, `build_nav`, `page_html`, and `highlight_loft` (syntax highlighting with inline stdlib links). |

`gendoc.rs` is the **orchestrator** â€” it owns the data-gathering loop and calls `documentation.rs` for all HTML emission. `documentation.rs` contains no file I/O of its own beyond writing the output files it is explicitly told to write.

## Syntax highlighting classes

| Class | Meaning |
|---|---|
| `kw` | Language keyword (`fn`, `if`, `for`, â€¦) |
| `ty` | Primitive type (`integer`, `text`, `vector`, â€¦) |
| `bi` | Built-in function (`assert`, `panic`, `len`, â€¦) |
| `en` | Capitalised identifier (struct/enum name) |
| `fn-call` | Identifier followed by `(` |
| `st` | String literal |
| `ch` | Character literal |
| `nm` | Numeric literal |
| `cm` | Line comment |

Identifiers whose name appears in the link map emit a `<a href="â€¦">` inside the span automatically.

## Adding a new topic page

1. Create `tests/docs/NN-name.loft` with `@NAME` and `@TITLE` headers.
2. Write comment blocks before each code group to explain what the code demonstrates.
3. Run `cargo run --bin gendoc` to regenerate all HTML files.

The new page is picked up automatically; the nav and index card grid are rebuilt from the full directory listing on every run.

## Adding documentation to the standard library

Add `//` doc comments immediately above `pub` declarations in `default/*.loft`.
Use `// --- Section Name ---` lines to group related items under a heading.
Run `cargo run --bin gendoc` to regenerate the section pages and update the link map.

If a new section is added, a new `doc/stdlib-<id>.html` file is created automatically and the section appears in the nav and search index on the next run.
