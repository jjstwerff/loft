# Phase 1.5 — Shared rayon work-stealing pool (native runtime)

**Status: open**

## Goal

Replace the `thread::scope`-per-call thread spawning in
`src/parallel.rs` with a single shared rayon work-stealing pool
that's initialised once per program lifetime.  Each `par(...)`
call submits tasks to the pool instead of spawning fresh OS
threads.

Matches the browser's existing `wasm-bindgen-rayon` model
(plan-06 phase 8); makes nested `par(...)` scale beyond ~3 levels
of nesting on native targets.

## Why now

D8 R2 allows nested par.  D2.1.1 shows the data-sharing
mechanism scales recursively at <10 µs per nested call.  But the
**thread-spawning** cost on `thread::scope` does not:

- Outer par spawns M threads.
- Each outer worker calling inner par spawns M more.
- At nesting depth K with M-way fan-out: M^K OS threads.
- M=4, K=2 → 16 threads (manageable).
- M=4, K=3 → 64 threads (scheduling overhead dominates).
- M=4, K=4 → 256 threads (OS thread-pool exhaustion likely).

A rayon-style work-stealing pool decouples task submission from
thread spawning: tasks queue onto a fixed-size pool (typically
`num_cpus`), regardless of nesting depth.  A nested par's inner
tasks are submitted onto the same pool already running outer
tasks; rayon's work-stealing balances them.

This makes nested par cost-bounded by **work**, not by
thread-spawn count.

## Why not earlier or later

Phase 1.5 sits between phase 1 (per-worker output stores) and
phase 2 (rebase) because:

- **After phase 1**: phase 1 establishes the per-worker
  WorkerStores construction shape; phase 1.5 just changes how
  the worker closures get scheduled, not what they receive.
- **Before phase 2**: phase 2's rebase walk is naturally
  per-call, independent of scheduler.  No interaction.
- **Independent of phases 3–9**: phase 1.5 is purely a runtime
  scheduler swap; phases 3–9 don't depend on which scheduler
  runs.  Could ship in any milestone window.

The phase is **small** (XS–S effort) — rayon is already a
dependency via `wasm-bindgen-rayon` (which depends on rayon
proper).  No new dependencies.

## What changes

### Today's mechanism (`src/parallel.rs::run_parallel_direct` etc.)

```rust
#[cfg(feature = "threading")]
std::thread::scope(|s| {
    for t in 0..threads {
        s.spawn(|| { /* worker body */ });
    }
});  // implicit join at scope exit
```

Each call spawns `threads` fresh OS threads.  Cost ~50 µs per
spawn on Linux; ~200 µs total per call at threads=4.  Plus OS
scheduling overhead for many concurrent threads.

### After phase 1.5

```rust
// One-time at program start
static POOL: OnceLock<rayon::ThreadPool> = OnceLock::new();
let pool = POOL.get_or_init(|| {
    rayon::ThreadPoolBuilder::new()
        .num_threads(num_cpus::get())
        .build()
        .expect("rayon pool init")
});

// Per par call
pool.scope(|s| {
    for t in 0..threads {
        s.spawn(|_| { /* worker body */ });
    }
});  // scope exit waits for all spawned tasks
```

Per-call cost drops from ~200 µs (thread spawn) to ~5 µs (task
submission).  Nested calls submit onto the same pool — no
thread-count explosion.

### What stays the same

- Worker fn body unchanged — it still receives a WorkerStores
  + WorkerOutputSlot.
- `WorkerStores` construction unchanged.
- D2.1.1 nested-par mechanism unchanged.
- Phase 2 rebase walk unchanged.
- Public API and user-visible behaviour unchanged.

## Per-commit landing plan

### 1.5a — add rayon dependency, init pool

- Add `rayon` to `Cargo.toml` `[dependencies]` (no feature gate;
  rayon is already transitively pulled in by `wasm-bindgen-rayon`
  on the wasm path).
- Add `parallel::POOL: OnceLock<rayon::ThreadPool>` initialised
  on first par call.
- No call-site changes yet; pool is unused.

Acceptance: `cargo build` + `make ci` green; `--no-default-features`
build still works (rayon has a small `no_std` path; or feature-gate
the pool init under `threading`).

### 1.5b — switch `run_parallel_direct` to pool.scope

- Replace `std::thread::scope` with `POOL.get().unwrap().scope` in
  `run_parallel_direct`.
- `s.spawn(|_| { ... })` instead of `s.spawn(|| { ... })` — rayon's
  scope passes a `&Scope` to closures.
- Bench: `bench/11_par`'s loft-interp + loft-native columns
  measured before / after; expect modest improvement at
  threads=4 (~150 µs/call savings) and large improvement on
  high-frequency par calls.

Acceptance: `make ci` green; `bench/11_par` within ±5 % of
phase-1 baseline; nested-par fixture (added in 1.5d) shows
sub-linear scaling vs thread::scope.

### 1.5c — switch other `run_parallel_*` variants

- Apply the same swap to `run_parallel_raw`, `run_parallel_text`,
  `run_parallel_ref`, `run_parallel_int`, `run_parallel_light`.
- All five variants now share the pool.

Acceptance: phase-0 characterisation suite green; bench-2/bench-3
within ±5 %.

### 1.5d — nested-par fixture + scaling test

- Add `tests/issues.rs::par_phase1_5_nested_depth_4_no_thread_explosion`
  — synthetic nested par at depth 4 (M=4 → 256 task units, but
  pool has `num_cpus` threads).  Asserts the test completes in
  <1 second and OS thread count never exceeds `num_cpus + 4`
  (rayon main + worker threads + overhead).
- Add `tests/issues.rs::par_phase1_5_pool_reused_across_calls`
  — runs 100 sequential par calls; asserts the pool is
  initialised exactly once (verify via a `static AtomicUsize`
  counter incremented in the OnceLock initialiser).

Acceptance: both fixtures green; `LOFT_LOG=minimal` shows pool
init message exactly once for the 100-call test.

### 1.5e — WASM unification (optional, defer to phase 8)

The browser path already uses `wasm-bindgen-rayon`.  Phase 8
formalises the COOP/COEP deployment.  Phase 1.5 doesn't change
the WASM path — it brings the native path to parity.

If phase 8 lands first, phase 1.5 is a pure native-only change.
If phase 1.5 lands first, phase 8 inherits the unified
abstraction (one rayon pool, two backends).

## Risks

| Risk | Mitigation |
|---|---|
| `rayon` brings transitive dependency growth | Already pulled in via `wasm-bindgen-rayon`; net new dependency cost is zero |
| Pool init failure on first par call | `OnceLock::get_or_init` propagates panic; covered by phase-0 panic-propagation fixture |
| `num_cpus` on WASM | Use `rayon::current_num_threads()` which the browser pool has already initialised; native uses `num_cpus::get()` |
| Pool size doesn't match user's `threads` arg | The `threads` arg is the **task count** to submit, not the pool size.  Pool size is `num_cpus` (or `LOFT_THREADS` env override).  Documented in phase-1.5e doc commit. |

## Out of scope

- **Per-call thread count guarantees** — today's `par(items, fn, 4)`
  spawns exactly 4 threads.  After 1.5, it submits 4 task chunks
  to the pool which may run on fewer threads if other work is
  active.  The user's `threads` arg becomes a **chunking hint**,
  not a thread-count guarantee.  Documented; tests updated.
- **Custom pool configuration per par call** — every par call uses
  the same global pool.  Per-call pool override is a future
  enhancement (1.0+).
- **Cancellation / abort** — rayon supports it; not adopted in
  plan-06 scope.  Worker panic propagation works as today
  (rayon's `scope` propagates panics to the parent thread on
  scope exit, matching `thread::scope`'s behaviour).

## Acceptance criteria

- `make ci` green at every sub-commit.
- `bench/11_par` loft-interp + loft-native columns within ±5 %
  of phase-1 baseline (small improvement expected from cheaper
  task submission).
- Nested par at depth 4 with M=4 fan-out completes in <1 s and
  OS thread count stays ≤ `num_cpus + 4`.
- Pool initialised exactly once across 100 sequential par calls.
- `LOFT_STORES=warn` reports zero leaked stores.

## Hand-off to phase 2

Phase 2 is unaffected — rebase walks worker output Stores
regardless of how those workers were scheduled.  Phase 1.5 ships
independently of phase 2's milestone.
