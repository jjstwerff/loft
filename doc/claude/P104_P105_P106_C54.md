## P104 — Direct struct return with vector literal

**Effort: S (2 steps)**

### Root cause (refined)

The `work_refs()` function (`src/variables/mod.rs:872-888`) creates temporary
variables named `__ref_1`, `__ref_2`, etc.  The counter `work_ref` resets to
zero for each function via `append()` (line 182-191).  In the flat namespace,
`__ref_1` from function A gets overwritten by function B's `__ref_1` when
the `names` HashMap is replaced during second-pass variable restoration.

When the Block codegen path processes `Value::Block` containing a struct
construction, it validates the work variable's type reference.  If the type
was set by function A but the variable now points to function B's definition,
`data.def()` receives u32::MAX → "Unknown definition".

The **Insert path** avoids this because it generates operations inline
without scope/frame management that triggers type validation on variables.

### Why the previous fix attempt failed

Setting `new_object = false` when `code` is `Value::Null` made the code
use `Value::Insert`, but Insert returns `Type::Rewritten(...)` instead of
`Type::Reference(...)`.  Functions returning structs need the actual
reference type, which only the Block path provides (via the last
`Value::Var(w)` in the Block's operator list).

### Fix: P104.1 — Make work variable names unique per function

**File:** `src/variables/mod.rs`, `work_refs()` function

The fix: include the function name or a unique function ID in the work
variable name to prevent flat-namespace collisions:

```rust
pub fn work_refs(&mut self, tp: &Type, lexer: &mut Lexer) -> u16 {
    // P104: include function context in name to avoid flat-namespace
    // collisions between functions that both create __ref_N variables.
    let n = format!("__ref_{}_{}", self.name, self.work_ref + 1);
    self.work_ref += 1;
    // ... rest unchanged ...
}
```

Where `self.name` is the current function's name (already available in
`Function`).  This makes `__ref_mat4_identity_1` distinct from
`__ref_mat4_mul_1`, preventing the HashMap collision.

**Also apply to:** `work_text()` and `work_rref()` if they have the same
naming scheme.

**Risk:** Low.  Work variables are internal compiler temporaries — changing
their names only affects the variable table, not generated bytecode.

### Fix: P104.2 — Regression tests

Remove the workaround from `mat4_identity()` (use direct return) and verify
all math tests pass.  Add a test to `76-ignored-struct-vector-return.loft`
for direct struct return with vector literal.

## P105 — Nested struct field access on vector elements

**Effort: S (2 steps)**

### Root cause (refined)

`OpGetVector` (`src/vector.rs:247-280`) returns a **synthetic inline DbRef**:
```
DbRef { store_nr, rec: v_rec, pos: 8 + (index * element_size) }
```

Where `v_rec` is the vector's storage record and `pos` is the byte offset
of the element within that record.

For base types (text, integer, etc.), `parse_vector_index` calls `get_val()`
which emits `OpGetText`/`OpGetInt` etc. — these read directly from the
inline position.  **Works correctly.**

For struct types (Reference), `get_val()` (`src/parser/mod.rs:1297-1303`)
dispatches to `OpGetRef` when the code is a `Value::Call` (which
`OpGetVector` is):

```rust
Type::Reference(_, _) => {
    if let Value::Call(_, _) = code {
        self.cl("OpGetRef", &[code, p])   // ← Dereferences through pointer
    } else {
        self.cl("OpGetField", &[code, p, ...])  // ← Simple offset addition
    }
}
```

**OpGetRef** (`src/fill.rs:1349-1354`) reads a pointer from the record,
following an indirection chain.  This fails for inline vector data because
the struct is stored inline (no pointer — just flat bytes).

**OpGetField** (`src/fill.rs:1368-1377`) simply adds the field offset to
`pos`.  This WOULD work correctly for inline vector elements, because:
```
result.pos = inline_pos + field_offset
```

### Fix: P105.1 — Use OpGetField for struct-typed vector elements

**File:** `src/parser/mod.rs`, `get_val()` function around line 1297

The condition `if let Value::Call(_, _) = code` was meant to handle
OpGetVector results, but OpGetRef is wrong for inline data.  The fix:
when accessing a Reference field on a value that came from OpGetVector,
use OpGetField instead of OpGetRef.

The simplest approach: after `parse_vector_index` returns for struct
elements, wrap the result so that subsequent field access uses the
non-Call path (OpGetField):

**File:** `src/parser/fields.rs`, `parse_vector_index()` around line 361

```rust
*code = self.cl("OpGetVector", &[code.clone(), Value::Int(elm_size), p]);
if self.database.is_base(known) || self.database.is_linked(known) {
    *code = self.get_val(etp, true, 0, code.clone());
} else if matches!(etp, Type::Reference(_, _)) {
    // P105: For struct elements, wrap OpGetVector result in OpGetField
    // with offset 0 so it becomes a non-Call Value.  This makes
    // subsequent field access use OpGetField (offset addition) instead
    // of OpGetRef (pointer dereference).
    *code = self.cl("OpGetField", &[code.clone(), Value::Int(0), self.type_info(etp)]);
}
```

`OpGetField` with offset 0 is a no-op for the position (just copies the
DbRef), but critically it makes `code` a `Value::Call("OpGetField", ...)`
instead of `Value::Call("OpGetVector", ...)`.

Wait — that doesn't help because `Value::Call` is still `Value::Call`.

**Better approach:** The real fix is in `get_val()` itself.  The condition
should check whether the Call is OpGetVector specifically:

```rust
Type::Reference(_, _) => {
    // P105: OpGetVector returns inline DbRef — use OpGetField (offset
    // addition), not OpGetRef (pointer dereference).
    let is_vector_element = matches!(&code, Value::Call(op, _) if {
        let name = /* extract op name */;
        name == "OpGetVector"
    });
    if is_vector_element {
        self.cl("OpGetField", &[code, p, self.type_info(tp)])
    } else if let Value::Call(_, _) = code {
        self.cl("OpGetRef", &[code, p])
    } else {
        self.cl("OpGetField", &[code, p, self.type_info(tp)])
    }
}
```

This keeps OpGetRef for normal Call results (e.g., function returns) but
uses OpGetField for OpGetVector results where the data is inline.

### Fix: P105.2 — Regression tests

Add to `tests/scripts/76-ignored-struct-vector-return.loft`:

```loft
struct Inner { ix: float not null, iy: float not null }
struct Outer { items: vector<Inner> }

fn test_nested_vector_field() {
  svo = Outer { };
  svo.items += [Inner { ix: 3.0, iy: 4.0 }];
  assert(svo.items[0].ix == 3.0, "nested field access");
}
```

**Risk:** Low-Medium.  Only changes the dispatch for OpGetVector results
in `get_val()`.  Normal struct field access (not from vector) is unaffected
because those don't go through OpGetVector.

**Effort: S (1 step, after P104 + P105)**

### Root cause

Compound effect of P104 (stale type references) and P105 (incorrect
field dereference for vector elements).  Likely resolves when both are
fixed.

Restore the full `test_scene_with_node` test with Mat4 assignment and
`cube()` mesh.  If store corruption persists, the root cause is in
