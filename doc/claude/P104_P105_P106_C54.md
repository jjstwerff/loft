## P104 — Test runner executing library functions — FIXED

### Root cause

The `--tests` runner treated ALL zero-parameter functions as test entry
points, including library functions loaded via `use`.  When it tried to
execute `mat4_identity()` (defined in `math.loft`), `data.def_nr` failed
because the function was registered under a different source.

### Fix

`src/test_runner.rs:651` — added source file filter:
`if def.position.file != abs_file { continue; }`.

## P105 — Nested struct field access on vector elements — OPEN

### Root cause

`get_val()` (`src/parser/mod.rs:1301-1307`) emits `OpGetRef` for all
`Value::Call` nodes accessing `Type::Reference` fields.  `OpGetRef` reads
4 bytes at `db.pos + fld` and treats the value as a record pointer.

Vector elements are stored inline (`src/vector.rs:247-280`).  Struct fields
within the element — including Reference fields like `inner: Inner` — are
stored as flat data at byte offsets.  `OpGetRef` reads this inline data as
a pointer, producing garbage (e.g., the integer 42 as a record number).

### Type system evaluation

The compiler's type system already has the information to make the correct
decision — it's just not used at the right place:

| Signal | Meaning | Location |
|---|---|---|
| `Parts::Vector(c)` | Elements inline at `8 + i*size` | `types.rs:68` |
| `Parts::Array(c)` | Elements are 4-byte pointers | `types.rs:69` |
| `Type.linked = true` | Type used in Hash/Index — pointer access | `types.rs:862` |
| `is_base(tp)` | Simple value type | `types.rs:151` |
| `is_linked(tp)` | Pointer-referenced type | `types.rs:146` |

The iterator code in `collections.rs:93-112` uses `is_linked` correctly
to choose between `OpVectorRef` (pointer) and `OpGetVector` (inline).
But `parse_vector_index` and `get_val` do not.

### Why parser-level fixes failed (3 attempts)

Changing which opcode `get_val` emits for OpGetVector results caused
intermittent "Unknown record" regressions in `tests/docs/16-parser.loft`.
The regressions were NOT caused by wrong logic — the P105 check was correct
for the specific case.  The regressions came from changing bytecode
generation globally, which had unpredictable side effects on the parser
library's internal store state when multiple files were loaded together.

### Why only `get_ref()` is vulnerable

`OpGetVector` returns `DbRef{rec=v_rec, pos=8+i*size}`.  The `v_rec` IS a
valid claimed record (the vector storage record).  All store operations that
read/write at `db.rec, db.pos + fld` work correctly — they access valid
positions within a valid record.

The **only** vulnerability is `get_ref()`, because it interprets the
**value read** as a **record number**:

| Operation | What it does with value | Safe? |
|---|---|---|
| `get_int/long/float/single/byte` | Returns value as data | **Safe** |
| `get_text` | Reads as text pointer | **Safe** — text fields always store pointers |
| `set_*` (all) | Writes data to position | **Safe** — writes to valid record at valid pos |
| **`get_ref`** | **Reads value as record number** | **Vulnerable** — inline struct data ≠ pointer |
| `set_ref` | Writes record number to position | **Safe** — only called for pointer-typed fields |

Why other operations are safe:
- **`db.rec`** (the vector storage record) is always in `claims` — store
  access never fails on the record itself
- **`db.pos + fld`** is a valid offset within that record — the store
  correctly reads/writes bytes at that position
- **Primitive values** (int, float, text pointers) are stored the same way
  in inline data as in normal records — the bytes are correct
- **Text pointers** are always record numbers even in inline structs —
  text is never stored inline, always as a pointer to string data
- **Vector pointers** are always record numbers even in inline structs —
  vector storage is separate from the struct data

**`get_ref` is unique** because it takes what might be inline struct data
(e.g., `{val: 42}` → the bytes `0x2A000000`) and treats those bytes as
a record number.  No other operation reinterprets data as a pointer.

### Recommended fix: runtime validation in `get_ref()`

**File:** `src/database/structures.rs:750`

```rust
pub fn get_ref(&self, db: &DbRef, fld: u32) -> DbRef {
    if db.rec == 0 {
        return DbRef { store_nr: db.store_nr, rec: 0, pos: 0 };
    }
    let store = self.store(db);
    let res = store.get_int(db.rec, db.pos + fld) as u32;
    // P105: validate the value against the store's claims.
    // If it's not a valid record, the data is inline — fall back
    // to offset addition (get_field behavior).
    if res == 0 || !store.is_claimed(res) {
        return DbRef {
            store_nr: db.store_nr,
            rec: db.rec,
            pos: db.pos + fld,
        };
    }
    DbRef { store_nr: db.store_nr, rec: res, pos: 8 }
}
```

### Why this fix is necessary and sufficient

**Necessary:** `get_ref` is the only operation that reinterprets data as
a record pointer.  All other store operations work correctly with inline
vector DbRefs because `db.rec` is a valid claimed record.

**Sufficient:** Once `get_ref` correctly detects inline data, the returned
DbRef points to the inline position.  Subsequent operations (get_int,
get_float, get_text, etc.) on that DbRef work correctly because they access
`db.rec` (still the valid vector record) at `db.pos + field_offset`.

### Why runtime validation is robust

1. **No bytecode changes** — same opcodes for all code paths
2. **No parser changes** — `get_val` stays as-is
3. **Authoritative** — `claims` is ground truth for allocated records
4. **O(1) cost** — single `HashSet::contains` per `OpGetRef` call
5. **Self-correcting** — works even if the compiler has other type bugs
6. **Targeted** — only `get_ref` needs the check; all other ops are safe

### Edge case: coincidental record match

Inline data could coincidentally equal a valid record number.  Mitigations:

- Record numbers are sequential from small values; inline struct data
  containing floats or larger integers is unlikely to collide
- Secondary check: verify the pointed-to record has a positive header
  (`size > 0`; freed records have negative headers)
- Long-term: the parser could be enhanced to emit `OpGetField` for
  known inline contexts, eliminating the runtime check entirely

### Store.claims accessibility

`claims` is currently private.  Add a public accessor:

```rust
// src/store.rs
pub fn is_claimed(&self, rec: u32) -> bool {
    self.claims.contains(&rec)
}
```

### Effort: XS

Two changes: `is_claimed()` method on Store (~3 lines), claims check
fallback in `get_ref()` (~5 lines changed).

## P106 — Store corruption with deep nested struct assignments — OPEN

### Root cause

Compound effect of P105.  `vector_add/copy_claims` deep-copies vector
elements correctly — vector and text pointers within inline structs ARE
valid record numbers (vectors and text are never stored inline).  The
corruption occurs when `OpGetRef` misinterprets inline struct data during
the field assignment chain leading up to the vector append.

### Fix

Retest after P105 `get_ref()` fix.  The deep-copy mechanism itself is
correct; the issue is in the access path that feeds data to it.
