## P105 ‚Äî Nested struct field access on vector elements ‚Äî FIXED

### Root cause

`get_val()` (`src/parser/mod.rs`) emitted `OpGetRef` for all `Value::Call` +
`Type::Reference` field accesses.  `OpGetRef` reads 4 bytes at `db.pos + fld`
and treats them as a record pointer.  But for inline structs in plain vectors,
those bytes are actual field data ‚Äî not a pointer.

Additionally, for **linked** struct types (struct used in both `vector<T>` and
`hash<T[key]>`, so the vector becomes `array<T>` storing 4-byte record pointers),
the `elm_size` passed to `OpGetVector` was the full struct size (e.g. 8) instead
of 4, causing wrong element positions for index > 0.

### Fix

**`src/parser/fields.rs` ‚Äî `parse_vector_index()`** (two call sites):

- **Linked structs** (`is_linked`): emit `OpVectorRef(vec, index)` directly.
  `OpVectorRef` internally uses elm_size=4 and dereferences the record pointer ‚Äî
  correct for all indices regardless of struct size.
- **Base/primitive types** (`is_base`): keep `OpGetVector(elm_size) + get_val()`
  to read the inline primitive value.
- **Plain inline structs** (neither): keep `OpGetVector(elm_size)` only; field
  access at the next `.` level uses `get_val` with the field offset.

**`src/parser/mod.rs` ‚Äî `get_val()` `Type::Reference` branch**:

Always emit `OpGetField` (offset addition).  The linked-type pointer dereference
is now handled entirely by `OpVectorRef` at the `fields.rs` call site.

- `tests/scripts/76-ignored-struct-vector-return.loft::test_p105_inline_struct_in_vector`
- `tests/scripts/76-ignored-struct-vector-return.loft::test_p105_nested_struct_in_vector`
- `tests/docs/16-parser.loft` ‚Äî full suite including linked `vector<Definition>` access

## P106 ‚Äî Nested vector in vector element had zero length ‚Äî FIXED

### Root cause

Same as P105.  `get_val()` emitted `OpGetRef` when reading an inline
`Type::Reference` field (e.g., `inner: Inner`) on a vector element.  This
caused reads from a wrong address, silently returning an empty-vector reference.

The symptom looked like a write-path copy bug but was actually a read-path
opcode bug ‚Äî the data was written correctly; the read returned garbage.

### Fix

Identical to P105.  `OpVectorRef` for linked types; `OpGetField` for all
`Type::Reference` accesses in `get_val()`.

### Test

- `tests/scripts/76-ignored-struct-vector-return.loft::test_p106_nested_vector_in_vector_element`
