## P105 — Nested struct field access on vector elements

### Root cause

`get_val()` emits `OpGetRef` for all `Value::Call` + `Type::Reference`.
`OpGetRef` reads 4 bytes and treats them as a record pointer.  Vector
elements are inline — the bytes are struct data, not a pointer.

### Current state: runtime fallback (Phase 1 complete)

`get_ref()` checks `store.is_claimed(res)`.  If false, falls back to
offset addition.  Works correctly but adds one `HashSet::contains` per
`OpGetRef` call.

### Phase 2: parser emits correct opcode — DEFERRED

**Goal:** Parser emits `OpGetField` for inline vector element access,
eliminating the runtime check.

**Status:** Four attempts failed with intermittent regressions in
`tests/docs/16-parser.loft`.  The regressions come from code paths in
`lib/code.loft` and `lib/parser.loft` where:

1. `vector<Definition>` stores pointers (elm_size=4) — OpGetRef is correct
2. The parser correctly distinguishes this via elm_size, but the test still
   fails intermittently, suggesting the bytecode change affects execution
   ordering or store state in ways that depend on file loading order

**Attempted checks:**
- `Value::Call(d, _) if *d == OpGetVector_dnr` — intermittent failures
- Same with `!self.first_pass` guard — same failures
- Same with `source_nr(0, "OpGetVector")` lookup — same failures
- Same with elm_size != 4 check — same failures

**Root cause unknown.** The parser change is logically correct for P105
cases but has side effects on unrelated code when multiple files are loaded
together.  Needs investigation with bytecode dump comparison (LOFT_LOG=static)
between working and failing runs.

**For now:** Phase 1 runtime fallback is the stable fix.  The HashSet
overhead is acceptable — `OpGetRef` is only called for Reference-typed
field access, which is relatively rare in hot loops.

### Phase 3: convert runtime fallback to debug_assert

**Prerequisite:** Phase 2 validation passes with zero fallback triggers
across 10 consecutive full test runs.

**File:** `src/database/structures.rs:750`

Change `get_ref()` from:
```rust
// Current: runtime fallback (Phase 1)
if res == 0 || !store.is_claimed(res) {
    return DbRef { store_nr: db.store_nr, rec: db.rec, pos: db.pos + fld };
}
```

To:
```rust
// Phase 3: debug-only assertion
debug_assert!(
    res == 0 || store.is_claimed(res),
    "P105: get_ref read non-record value {res} at rec={} pos={}+{fld} \
     — inline vector data reached OpGetRef; parser should emit OpGetField",
    db.rec, db.pos
);
```

**Validation after Phase 3:**

1. Debug build tests (default `cargo test`):
   ```
   cargo test
   ```
   Expect: all pass.  The `debug_assert` fires in debug builds, so any
   remaining inline-data-reaching-OpGetRef would panic with a clear
   message instead of "Unknown record".

2. Release build tests:
   ```
   cargo test --release
   ```
   Expect: all pass.  The `debug_assert` is compiled out.  The parser
   (Phase 2) emits the correct opcode, so OpGetRef never receives
   inline data.

3. Performance validation — no regression in release builds:
   ```
   cargo run --release --bin loft -- --lib lib/graphics/src \
     --tests lib/graphics/tests/canvas.loft
   ```
   Expect: same timing as before the P105 fix (no HashSet overhead).

4. Run 10 consecutive times to confirm no intermittent issues:
   ```
   for i in 1..10; do cargo test 2>&1 | grep FAILED; done
   ```

### Phase 4 (optional): remove `is_claimed()` method

Once Phase 3 is stable for at least one release cycle, the `is_claimed()`
method on Store can be removed if no other code uses it.  The `debug_assert`
in `get_ref` can be simplified to use `store.claims.contains()` directly
if claims becomes `pub(crate)`.

### Edge case: coincidental record match

Inline data could coincidentally equal a valid record number.

**Phase 1–2 mitigation:** The runtime fallback checks claims.  A false
positive (inline value = valid record) would cause `get_ref` to dereference
instead of offsetting.  This is rare but could corrupt data silently.

**Secondary validation** (add in Phase 2):
```rust
if res != 0 && store.is_claimed(res) {
    // Secondary: verify the record has a positive header (not freed)
    let header: i32 = *store.addr(res, 0);
    if header <= 0 {
        // Freed record or corruption — treat as inline
        return DbRef { store_nr: db.store_nr, rec: db.rec, pos: db.pos + fld };
    }
}
```

**Phase 3 mitigation:** The parser emits the correct opcode, so the
coincidence path is never reached.  The `debug_assert` catches it in
test builds.

### Summary

| Phase | Change | Runtime cost | Safety |
|---|---|---|---|
| 1 (done) | `get_ref()` fallback | O(1) HashSet/call | Full — self-correcting |
| 2 | Parser OpGetVector check | Zero (correct opcode) | Full — Layer 1 fallback catches misses |
| 3 | Fallback → `debug_assert` | Zero in release | Debug builds catch regressions |

## P106 — Store corruption with deep nested struct assignments — OPEN

### Root cause

NOT a P105 effect.  The `fl_validate` error ("positive header 0") comes
from store allocation during vector append of a struct containing
`vector<float>` (Mat4).  The deep-copy mechanism in `copy_claims()` may
not handle nested vector fields correctly when the source struct is built
incrementally (field-by-field assignment before append).

### Reproducer

```loft
sn_n = scene::node("cube1", sn_mi, sn_mati);
sn_n.transform = math::mat4_translate(1.0, 0.0, 0.0);  // Mat4 has vector<float>
sn_s.add_node(sn_n);  // crashes: fl_validate on the appended node
```

Without `mat4_translate` (using default identity transform), the test passes.

### Next step

Investigate `copy_claims` in `src/database/structures.rs` for how it
handles struct fields that contain vectors.  The `vector<float>` inside
Mat4 may not be deep-copied correctly during the `add_node` vector append.
