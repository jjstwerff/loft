# Fix Design: P104, P105, P106, C54

Detailed fix designs for the four issues discovered during Sprint 8.

---

## C54 — Exponentiation `** 0.5` codegen crash

**Effort: XS (1 step)**

### Root cause

`**` is not in the OPERATORS table (`src/parser/mod.rs:148-160`).  The lexer
produces two `*` tokens.  The parser consumes the first as multiply, then
fails to parse a unary `*`, producing `Value::Null`.  Codegen asserts on the
null argument: "expected 8B on stack but generate(Null) pushed 0B".

### Fix: C54.1 — Add `**` operator mapped to `pow()`

**File:** `src/parser/mod.rs`

1. Add `"**"` to the OPERATORS table at a new precedence level **above** `*`
   (right-associative, like most languages):

```rust
static OPERATORS: &[&[&str]] = &[
    &["??"],
    &["||", "or"],
    &["&&", "and"],
    &["==", "!=", "<", "<=", ">", ">="],
    &["|"],
    &["^"],
    &["&"],
    &["<<", ">>"],
    &["-", "+"],
    &["*", "/", "%"],
    &["**"],           // NEW — exponentiation, higher precedence than *
    &["as"],
];
```

2. Add rename mapping in the `rename()` function (~line 2438-2460):

```rust
"**" => "pow",
```

3. The lexer must recognise `**` as a single token.  Check `src/lexer.rs` —
   the two-char operator matching should already handle this since `**` starts
   with `*` and the second char is `*`.  If not, add it to the two-char
   operator list.

**Test:** Remove `@EXPECT_FAIL` from `tests/scripts/77-ignored-exponentiation.loft`.

**Risk:** Low.  `pow()` already exists for both `float` and `single`.

## P104 — Direct struct return with vector literal

**Effort: M (3 steps)**

### Root cause

When a struct with a vector field is returned directly (no intermediate
variable), the parser wraps it in `Value::Block` via the `new_object` path
(`src/parser/objects.rs:1033-1040`).  This creates a work variable
`__ref_N` and generates `OpDatabase` inside a Block.

The Block code path in `generate_block` (`src/state/codegen.rs:1437-1503`)
manages stack frames differently from `Value::Insert`.  When many functions
are compiled in the same file, the work variable's type definition reference
can become stale (u32::MAX), causing "Unknown definition" in `data.def()`.

The `Value::Insert` path (lines 189-193) simply generates operations inline
without scope management, which avoids the stale reference problem.

### Why Block exists

The comment at `objects.rs:1020-1022` explains: Block is needed when a struct
construction appears as a **method-call argument**, because the temporary
allocation needs proper scope/frame semantics.  For direct returns and simple
assignments, Insert works fine.

### Fix steps

#### P104.1 — Detect direct-return context in struct construction

**File:** `src/parser/objects.rs`

Currently `new_object` is set whenever code is not a `Value::Var` and not a
field (line 1033).  Add a flag to distinguish "struct as return value" from
"struct as method argument":

```rust
// Line ~1033: instead of unconditionally setting new_object = true,
// check if we're in a return-expression context.
} else if !self.first_pass && !self.is_field(code) {
    // If code is Value::Null (fresh expression, not an argument),
    // we can safely use Insert instead of Block.
    if matches!(code, Value::Null) {
        // Direct return / assignment — use Insert path
        // Create a local variable but don't wrap in Block
        let ret = &self.data.def(td_nr).returned;
        let w = self.vars.work_refs(ret, &mut self.lexer);
        let tp = i32::from(self.data.def(td_nr).known_type);
        list.push(v_set(w, Value::Null));
        list.push(self.cl("OpDatabase", &[Value::Var(w), Value::Int(tp)]));
        *code = Value::Var(w);
        // new_object stays false → Insert path at line 1091
    } else {
        // Struct in expression context (method arg, etc.) — need Block
        new_object = true;
        // ... existing code ...
    }
}
```

**Test:** The existing `lib/graphics/src/math.loft::mat4_identity()` can be
changed back to direct return.  Run `lib/graphics/tests/math.loft` to verify.

#### P104.2 — Add regression test for direct struct return with vector

**File:** `tests/scripts/76-ignored-struct-vector-return.loft`

Change the workaround test to use direct return:

```loft
fn svr_identity_direct() -> SvMat {
  SvMat { vals: [1.0, 0.0, 0.0, 1.0] }
}

fn test_p104_direct_return() {
  svr_d = svr_identity_direct();
  assert(svr_d.vals.len() == 4, "direct return works");
}
```

#### P104.3 — Verify struct-as-method-argument still works

Ensure the Block path is still used for method arguments:

```loft
fn takes_mat(tm: SvMat) -> integer { tm.vals.len() }
fn test_struct_as_argument() {
  assert(takes_mat(SvMat { vals: [1.0, 2.0] }) == 2, "struct arg");
}
```

**Risk:** Medium.  The condition for choosing Insert vs Block must be precise.
If we accidentally use Insert for a method-argument context, stack corruption
can occur (P103-style).

## P105 — Nested struct field access on vector elements

**Effort: M (3 steps)**

### Root cause

`OpGetVector` returns a **synthetic DbRef** with `rec = vector_record` and
`pos = 8 + (i * element_size)` — an inline offset into the vector's storage.

For base types, `parse_vector_index` wraps the result with `get_val()` (line
362 in `fields.rs`), extracting the primitive value.  For struct types, the
OpGetVector result is left as-is.

When the next field access (`.pos`) is parsed, it calls `get_field()` which
dispatches to `OpGetRef` for Reference-typed fields.  `OpGetRef` expects a
proper struct record, but receives the vector-inline DbRef.  It produces an
invalid record number (0), triggering the store's "Unknown record" assertion.

**Working case:** `items[0].name` — `name` is a `text` (base type), so
`get_val()` extracts it correctly from the inline position.

**Broken case:** `vertices[0].pos.x` — `pos` is a `Vec3` (struct/Reference),
so `OpGetRef` is used, which doesn't handle inline vector positions.

### Fix steps

#### P105.1 — Add inline struct extraction after OpGetVector

**File:** `src/parser/fields.rs`, `parse_vector_index()` around line 361

After `OpGetVector`, when the element type is a struct (Reference), we need
to copy the inline struct data to a proper struct record that can be used for
subsequent field access:

```rust
// After line 363: add struct-element handling
*code = self.cl("OpGetVector", &[code.clone(), Value::Int(elm_size), p]);
if self.database.is_base(known) || self.database.is_linked(known) {
    *code = self.get_val(etp, true, 0, code.clone());
} else if matches!(etp, Type::Reference(_, _)) {
    // P105: struct elements need OpCopyInline to create a proper
    // record that subsequent field access can use.
    // For now, leave as-is — the DbRef from OpGetVector points
    // at the correct inline data.  The issue is that OpGetRef
    // doesn't handle inline positions.
}
```

The real fix is in `OpGetRef` — it should detect when its input is an inline
DbRef (from OpGetVector) and adjust its record/position calculation
accordingly.

**Alternative approach:** Add a new opcode `OpGetInlineStruct` that copies
inline struct data from a vector element to a fresh record, enabling standard
field access.  This is cleaner but requires a new opcode slot (253/256 used).

#### P105.2 — Fix OpGetRef for inline vector positions

**File:** `src/fill.rs` — the `OpGetRef` implementation

Currently `OpGetRef` reads a reference field from a record:
```rust
// Pseudo-code of current OpGetRef:
let db = stack.pop::<DbRef>();
let pos = param;
let ref_val = stores.store(&db).get_ref(db.rec, db.pos + pos);
stack.push(ref_val);
```

For inline vector elements, `db.rec` is the vector's storage record and
`db.pos` is the element offset.  The fix: when the input DbRef has a
non-zero `pos` that indicates inline data, compute the nested field position
as `db.pos + field_offset` rather than dereferencing through a record table.

This requires distinguishing "normal DbRef" from "inline DbRef".  Options:
- Check if `db.rec` is a vector record (has vector metadata)
- Add a flag bit to `db.pos` to mark inline references
- Track inline state in the codegen type system

#### P105.3 — Add regression tests

**File:** `tests/scripts/76-ignored-struct-vector-return.loft`

```loft
struct Inner { ix: float not null, iy: float not null }
struct Outer { items: vector<Inner> }

fn test_nested_vector_field() {
  svo = Outer { };
  svo.items += [Inner { ix: 3.0, iy: 4.0 }];
  assert(svo.items[0].ix == 3.0, "nested field access");
}
```

**Risk:** Medium.  Changing OpGetRef affects all reference field access.  Must
ensure normal (non-inline) struct field access is unaffected.  Extensive
testing required.

## P106 — Store corruption with deep nested struct assignments

**Effort: S (1 step, after P104 + P105)**

### Root cause

Assigning a Mat4 (struct with vector<float>) to a Node.transform field
inside a Scene.nodes vector triggers store validation failures.  This is a
compound effect of P104 (struct construction in nested context) and P105
(field access through vector elements).

The assignment path is:
1. `sn_n.transform = mat4_translate(...)` — assigns Mat4 to Node field
2. `sn_s.add_node(sn_n)` — appends Node (containing Mat4) to Scene.nodes vector
3. The vector append must copy the Mat4 including its inner vector<float>
4. The store's dependency tracking gets confused about which records are live

### Fix: P106.1 — Retest after P104 + P105

After fixing P104 and P105, rerun the simplified `test_scene_with_node` test
with the original complexity restored:

```loft
fn test_scene_with_node() {
  sn_s = scene::scene("complete");
  sn_mi = sn_s.add_mesh(mesh::cube());
  sn_mati = sn_s.add_material(scene::material("default"));
  sn_n = scene::node("cube1", sn_mi, sn_mati);
  sn_n.transform = math::mat4_translate(1.0, 0.0, 0.0);
  sn_s.add_node(sn_n);
  assert(sn_s.nodes.len() == 1, "one node");
  assert(sn_s.nodes[0].name == "cube1", "node name");
}
```

If it still fails, the root cause is in `src/store.rs` `fl_validate` — the
vector append operation for nested structs doesn't properly update the claims
set for copied inner records.

**Risk:** Low.  This is a retest, not a code change.
