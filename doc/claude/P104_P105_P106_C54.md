## P105 — Nested struct field access on vector elements

### Root cause

`get_val()` emits `OpGetRef` for all `Value::Call` + `Type::Reference`.
`OpGetRef` reads 4 bytes and treats them as a record pointer.  Vector
elements are inline — the bytes are struct data, not a pointer.

### Current state: runtime fallback (Phase 1 complete)

`get_ref()` checks `store.is_claimed(res)`.  If false, falls back to
offset addition.  Works correctly but adds one `HashSet::contains` per
`OpGetRef` call.

### Phase 2: parser emits correct opcode — DEFERRED

**Goal:** Parser emits `OpGetField` for inline vector element access,
eliminating the runtime check.

**Status:** Four attempts failed with intermittent regressions in
`tests/docs/16-parser.loft`.  The regressions come from code paths in
`lib/code.loft` and `lib/parser.loft` where:

1. `vector<Definition>` stores pointers (elm_size=4) — OpGetRef is correct
2. The parser correctly distinguishes this via elm_size, but the test still
   fails intermittently, suggesting the bytecode change affects execution
   ordering or store state in ways that depend on file loading order

**Attempted checks:**
- `Value::Call(d, _) if *d == OpGetVector_dnr` — intermittent failures
- Same with `!self.first_pass` guard — same failures
- Same with `source_nr(0, "OpGetVector")` lookup — same failures
- Same with elm_size != 4 check — same failures

**Root cause unknown.** The parser change is logically correct for P105
cases but has side effects on unrelated code when multiple files are loaded
together.  Needs investigation with bytecode dump comparison (LOFT_LOG=static)
between working and failing runs.

**For now:** Phase 1 runtime fallback is the stable fix.  The HashSet
overhead is acceptable — `OpGetRef` is only called for Reference-typed
field access, which is relatively rare in hot loops.

### Phase 3: convert runtime fallback to debug_assert

**Prerequisite:** Phase 2 validation passes with zero fallback triggers
across 10 consecutive full test runs.

**File:** `src/database/structures.rs:750`

Change `get_ref()` from:
```rust
// Current: runtime fallback (Phase 1)
if res == 0 || !store.is_claimed(res) {
    return DbRef { store_nr: db.store_nr, rec: db.rec, pos: db.pos + fld };
}
```

To:
```rust
// Phase 3: debug-only assertion
debug_assert!(
    res == 0 || store.is_claimed(res),
    "P105: get_ref read non-record value {res} at rec={} pos={}+{fld} \
     — inline vector data reached OpGetRef; parser should emit OpGetField",
    db.rec, db.pos
);
```

**Validation after Phase 3:**

1. Debug build tests (default `cargo test`):
   ```
   cargo test
   ```
   Expect: all pass.  The `debug_assert` fires in debug builds, so any
   remaining inline-data-reaching-OpGetRef would panic with a clear
   message instead of "Unknown record".

2. Release build tests:
   ```
   cargo test --release
   ```
   Expect: all pass.  The `debug_assert` is compiled out.  The parser
   (Phase 2) emits the correct opcode, so OpGetRef never receives
   inline data.

3. Performance validation — no regression in release builds:
   ```
   cargo run --release --bin loft -- --lib lib/graphics/src \
     --tests lib/graphics/tests/canvas.loft
   ```
   Expect: same timing as before the P105 fix (no HashSet overhead).

4. Run 10 consecutive times to confirm no intermittent issues:
   ```
   for i in 1..10; do cargo test 2>&1 | grep FAILED; done
   ```

### Phase 4 (optional): remove `is_claimed()` method

Once Phase 3 is stable for at least one release cycle, the `is_claimed()`
method on Store can be removed if no other code uses it.  The `debug_assert`
in `get_ref` can be simplified to use `store.claims.contains()` directly
if claims becomes `pub(crate)`.

### Edge case: coincidental record match

Inline data could coincidentally equal a valid record number.

**Phase 1–2 mitigation:** The runtime fallback checks claims.  A false
positive (inline value = valid record) would cause `get_ref` to dereference
instead of offsetting.  This is rare but could corrupt data silently.

**Secondary validation** (add in Phase 2):
```rust
if res != 0 && store.is_claimed(res) {
    // Secondary: verify the record has a positive header (not freed)
    let header: i32 = *store.addr(res, 0);
    if header <= 0 {
        // Freed record or corruption — treat as inline
        return DbRef { store_nr: db.store_nr, rec: db.rec, pos: db.pos + fld };
    }
}
```

**Phase 3 mitigation:** The parser emits the correct opcode, so the
coincidence path is never reached.  The `debug_assert` catches it in
test builds.

### Summary

| Phase | Change | Runtime cost | Safety |
|---|---|---|---|
| 1 (done) | `get_ref()` fallback | O(1) HashSet/call | Full — self-correcting |
| 2 | Parser OpGetVector check | Zero (correct opcode) | Full — Layer 1 fallback catches misses |
| 3 | Fallback → `debug_assert` | Zero in release | Debug builds catch regressions |

## P106 — Store corruption with deep nested struct assignments — OPEN

### Root cause

NOT a P105 effect.  The `fl_validate` error ("positive header 0") comes
from store allocation during vector append of a struct containing
`vector<float>` (Mat4).  The deep-copy mechanism in `copy_claims()` may
not handle nested vector fields correctly when the source struct is built
incrementally (field-by-field assignment before append).

### Minimal reproducer

```loft
struct Inner { vals: vector<float> }
struct Outer { name: text, inner: Inner }
struct Top { items: vector<Outer> }

fn main() {
  t = Top {};
  o = Outer { name: "x", inner: Inner { vals: [1.0, 2.0] } };
  println("before: {o.inner.vals.len()}");  // prints 2
  t.items += [o];
  println("after: {t.items[0].inner.vals.len()}");  // prints 0 — BUG
}
```

**Works:** `struct HasVec { vals: vector<int> }` in `vector<HasVec>` — direct
vector field survives the deep-copy.

**Fails:** `struct Outer { inner: Inner }` where Inner has a vector — nested
struct's vector has length 0 after append.

### Investigation findings

**Store layout:** `sizeof(Inner) = 4` (just a vector pointer).
`sizeof(Outer) = 8` (text pointer + Inner inline).  Inner is stored inline
in Outer, not as a separate record.

**Copy path:** `vector_add` (`structures.rs:326-412`) does:
1. Raw byte copy of element data (`copy_block`)
2. `copy_claims(rec, to, Outer_type)` for each element

`copy_claims` (`allocation.rs:622-675`) recurses through:
1. `Outer` → `Parts::Struct([name, inner])` → process each field
2. `name: text` → `Parts::Base` → deep-copy text string ✓
3. `inner: Inner` → `Parts::Struct([vals])` → recurse into Inner
4. `vals: vector<float>` → `Parts::Vector(float)` → `copy_claims_seq_vector`

`copy_claims_seq_vector` (`allocation.rs:396-437`):
- Line 399: `let cur = self.store(rec).get_int(rec.rec, rec.pos) as u32;`
- Reads the vector pointer from the SOURCE position
- If `cur == 0` → returns early, writes 0 to destination (empty vector)
- Otherwise → claims new space, copies vector data, writes new pointer

**Suspected cause:** The `claim()` call at line 404 may trigger store
growth (`claim_grow` → `resize_store`), which reallocates the store's
memory.  Since source and destination are in the SAME store, the source
`rec` DbRef becomes invalid after reallocation — subsequent reads of the
source vector pointer return garbage or 0.

This would explain why direct vector fields work (they're processed first,
before the store grows) but nested struct vectors fail (processed later,
after intervening claims may have triggered reallocation).

### Fix options

| Option | Effort | Risk | Description |
|---|---|---|---|
| **A. Pre-read source** | S | Low | Read vector pointer+length into locals before any claim call |
| B. Snapshot source bytes | M | Low | Clone entire source element into a temp buffer before copy_claims |
| C. Separate stores | M | Med | Copy to temp store, then move to destination |

### Recommended fix: Option A — pre-read source in `copy_claims_seq_vector`

**File:** `src/database/allocation.rs`, `copy_claims_seq_vector()` at line 396

The current code reads the source vector pointer BEFORE `claim()` (line 399),
which looks safe.  But recursive `copy_claims` calls inside the element loop
(lines 424-436) may trigger additional `claim()` calls that grow the store.

The fix: capture all source data into local variables before ANY writes.

```rust
pub(super) fn copy_claims_seq_vector(&mut self, rec: &DbRef, to: &DbRef, tp: u16) {
    let length = vector::length_vector(rec, &self.allocations);
    let size = u32::from(self.size(tp));
    let cur = self.store(rec).get_int(rec.rec, rec.pos) as u32;
    if cur == 0 {
        self.store_mut(to).set_int(to.rec, to.pos, 0);
        return;
    }
    // P106: snapshot source vector bytes BEFORE any claim() calls.
    // claim() may trigger store growth (resize_store), invalidating
    // source DbRefs when source and destination share the same store.
    let src_bytes = self.store(rec).read_bytes(cur, 4, length * size + 4);
    // Now safe to allocate destination
    let into = self.store_mut(to).claim(1 + (size * length).div_ceil(8));
    self.store_mut(to).set_int(to.rec, to.pos, into as i32);
    self.store_mut(to).write_bytes(into, 4, &src_bytes);
    // Recursive copy_claims for elements (text, nested vectors, etc.)
    for i in 0..length {
        self.copy_claims(
            &DbRef { store_nr: rec.store_nr, rec: cur, pos: 8 + size * i },
            &DbRef { store_nr: to.store_nr, rec: into, pos: 8 + size * i },
            tp,
        );
    }
}
```

**Key change:** `read_bytes` captures the source vector's raw bytes into
a `Vec<u8>` before `claim()` is called.  Then `write_bytes` writes to
the destination.  This ensures the source is not invalidated by store growth.

**Note:** `read_bytes` / `write_bytes` don't exist yet on Store.  They're
simple wrappers around `std::ptr::copy_nonoverlapping`.

**Prerequisite:** Verify the reallocation theory first by adding a
`debug_assert` in `claim_grow` that checks whether any active `copy_claims`
operation is in progress.

### Alternative: apply same pattern to copy_claims for structs

The struct recursion in `copy_claims` (lines 637-652) may also be affected:
recursive `copy_claims` on nested struct fields may trigger claims that
invalidate the outer struct's source data.  If Option A fixes the vector
case but struct-of-struct cases still fail, extend the snapshot approach
to the struct loop as well.

### Effort: S–M

S if the vector snapshot alone fixes P106.  M if the struct recursion
also needs snapshotting.
