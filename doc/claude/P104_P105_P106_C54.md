## P104 — Test runner executing library functions — FIXED

### Root cause (actual)

The `--tests` runner treated ALL zero-parameter functions as test entry
points, including library functions loaded via `use`.  When it tried to
execute `mat4_identity()` (defined in `math.loft`, a library), `data.def_nr`
failed because the function was registered under a different source than
the test file's source.

The "Unknown definition" (u32::MAX) came from `execute_argv` at
`src/state/mod.rs:1212`, not from codegen.  The previous theories about
flat-namespace variable collisions and Block codegen were wrong.

### Fix

**File:** `src/test_runner.rs`, line 647

Added source file filter: `if def.position.file != abs_file { continue; }`.
Only functions defined in the test file itself are executed as tests.

**Test:** `tests/scripts/76-ignored-struct-vector-return.loft::test_p104_direct_return`.

## P105 — Nested struct field access on vector elements — OPEN

### Root cause

`get_val()` (`src/parser/mod.rs:1297-1307`) emits `OpGetRef` for all
`Value::Call` nodes when accessing `Type::Reference` fields.  `OpGetRef`
reads 4 bytes at `db.pos + fld` and treats the value as a record number
(pointer dereference).

`OpGetVector` returns an inline DbRef (`rec = vector_storage_rec`,
`pos = 8 + index * elem_size`).  Struct fields within the inline element
are stored as flat data at byte offsets — **not as record pointers**.
When `OpGetRef` reads the inline bytes as a record number, it gets garbage
(e.g., the integer value 42 instead of a valid record), causing
"Unknown record" crashes.

### Key finding: all vector elements are inline

Investigation confirmed that **all vector elements are stored inline**
regardless of whether the element type is "linked" or not:

- `vector::get_vector()` (`src/vector.rs:247-280`): returns `DbRef{rec: v_rec, pos: 8 + size*i}` — no pointer read
- `vector::vector_append()` (`src/vector.rs:80-109`): writes element data at `8 + pos*size` — inline copy
- Struct fields within vector elements: stored at flat offsets within the element's inline block
- Reference fields (e.g., `inner: Inner`): the Inner struct's data is stored inline within the parent struct's data block — NOT as a 4-byte pointer

### Why parser-level fixes failed

Three approaches were tried and all caused intermittent regressions:

1. **Check OpGetVector by def_nr comparison** — caused intermittent "Unknown record" in `tests/docs/16-parser.loft` when run through the test suite. The `lib/code.loft` library's `definitions: vector<Definition>` accesses were affected despite no Reference fields on Definition.

2. **Check OpGetVector by name comparison** — same intermittent failure.

3. **Wrap OpGetVector result in work variable** — "Use after free" because the work variable's scope ended before the field access chain completed.

The parser approach is fundamentally brittle because changing which opcode is emitted alters bytecode generation globally, with unpredictable interactions across files loaded together.

### Recommended fix: runtime fallback in `get_ref()`

**File:** `src/database/structures.rs`, `get_ref()` at line 750

Instead of changing the parser, make `get_ref()` detect when the value it
reads isn't a valid record and fall back to `get_field()` behavior:

```rust
pub fn get_ref(&self, db: &DbRef, fld: u32) -> DbRef {
    if db.rec == 0 {
        return DbRef { store_nr: db.store_nr, rec: 0, pos: 0 };
    }
    let store = self.store(db);
    let res = store.get_int(db.rec, db.pos + fld) as u32;
    // P105: if the value read doesn't look like a valid record,
    // the data is inline (e.g., struct field within a vector element).
    // Fall back to offset addition (get_field behavior).
    if res == 0 || !store.claims.contains(&res) {
        return DbRef {
            store_nr: db.store_nr,
            rec: db.rec,
            pos: db.pos + fld,
        };
    }
    DbRef { store_nr: db.store_nr, rec: res, pos: 8 }
}
```

**Why this is robust:**
- No bytecode changes — same opcodes emitted for all code paths
- No parser changes — get_val stays exactly as-is
- Self-healing — if OpGetRef is called on inline data, it detects and adjusts
- Zero cost for correct cases — `claims.contains()` is O(1) HashSet lookup
- The `claims` set tracks all allocated records; an inline data value (e.g., 42)
  will never be in claims unless it coincidentally matches an allocated record

**Risk:** If an inline value coincidentally equals a valid record number, the
fallback would not trigger and the bug would still occur. This is unlikely
in practice because record numbers are sequential from small values while
inline struct data is often larger or floating-point.

**Effort:** XS — single function change, no new opcodes.

## P106 — Store corruption with deep nested struct assignments — OPEN

### Root cause

Compound effect of P105.  Assigning a Mat4 (struct with vector<float>) to
a Node field triggers store corruption because nested struct operations
chain through OpGetRef on inline data.

### Fix

Retest after P105 is fixed.  If it still fails, investigate
`src/store.rs` `fl_validate` for vector append of nested structs.
